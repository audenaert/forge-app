## Description

A mechanism for running a candidate task-agent variant *alongside* the production variant on real customer traffic, but only serving the production variant's result to the customer. The candidate's execution — its tokens, its tool calls, its structural outputs, its quality proxies, and (ephemerally) its actual generated content for comparison purposes — is analyzed. The customer never sees the candidate; the candidate never affects the customer. But Etak gets to see how the candidate would have behaved on exactly the workloads it would actually face in production, and build confidence (or destroy it) before any customer is exposed.

Shadow execution sits *before* the experimentation harness (idea E) in the promotion pipeline, not alongside it. The path from "someone has a candidate agent version" to "customers see it in production" looks something like:

1. **Synthetic evals.** Does the candidate pass the curated test suite? If not, stop.
2. **Shadow execution.** Does the candidate behave reasonably on live traffic when the customer isn't watching? If it crashes, hits guardrails, or looks structurally or substantively worse than production on real inputs, stop.
3. **Live experimentation.** Does the candidate actually win or lose against production when real customers see its output? If the harness says "inconclusive" or "production wins," stop.
4. **Promotion decision.** Human reviews the evidence and decides whether to ship.

Shadow is the step that catches the candidates that would have *failed* the experimentation phase in ways that harm customers. It's the difference between "let's run this on 5% of traffic and see what happens" and "let's run this invisibly first so we know whether 5% is safe."

## Strategic Rationale

The shadow runner is the front line of the customer-level safety bar. Without it, Etak is only left with two options for every candidate:

- **Ship it to an experiment directly.** Which means real customers experience the candidate's failures before anyone has seen them — which violates the opportunity's "any individual customer" safety commitment for a category of failures that were avoidable.
- **Test it only on synthetic evals.** Which means Etak is deciding whether a candidate is safe based on data that doesn't look like the real long tail of customer workloads. Evals catch the obvious failures but miss the ones that only appear against the messy reality of actual usage.

Shadow mode closes the gap between these two. It gives Etak a way to see a candidate behave on real traffic *without* the risk of real customer exposure. This is the specific tool for the specific risk the experimentation opportunity calls out: that the first customer to see a regression shouldn't be the one who proves the regression existed.

There's a secondary rationale around cost and cadence. Experimentation is expensive in a different way than shadow mode — experiments require statistical power, cohort management, guardrail monitoring, and real-time rollback. Shadow mode is cheaper per candidate (it's just running inference in parallel and analyzing the results) and can be applied to many more candidates than experimentation can. This lets Etak pre-filter candidates down to a smaller set that's actually worth the cost of a live experiment. Without shadow, every candidate either goes to experiment (expensive) or doesn't get tested at all (risky).

## How It Could Work

- **Shadow is a property of a task-agent version, not a separate system.** When a new version of a task-agent is staged in the registry (idea B), a "shadow against production" flag can be set on it. The gateway (idea A), when it sees a request for that task, executes *both* the production version (serving its result) and the shadowed candidate (discarding its result but analyzing everything about the run). To the calling code, it's a single call; the dual execution is a gateway concern.
- **Shadow runs are tagged and emit the full telemetry schema.** Every shadow execution produces a run event (idea C) with a `shadow=true` flag and a pointer to the corresponding production run. This lets the analysis layer pair up production and shadow runs for direct comparison — same input, two outputs, two sets of quality proxies.
- **Ephemeral content-aware comparison is the default.** Shadow mode's most valuable signal comes from comparing the candidate's output to production's. This comparison is allowed because the privacy constraint is about retention, not processing. Embedding similarity over the two outputs, an LLM judge scoring both against the original request, structural divergence measures, error-mode diffing — all of these run on the content of both runs and are allowed to do so, as long as the content itself is not retained beyond the comparison step. The signals that come *out* of the comparison (similarity score, judge rating, structural divergence summary) are retained as part of the telemetry record; the inputs and outputs are not.
- **Retention is explicit and narrow.** The shadow's output exists in memory only long enough to compute the comparison signals, then is discarded. The production output is retained only on the same terms as any non-shadow production run — which means it is not retained beyond the customer session unless the customer is in the data partnership tier. What persists from a shadow run is: the run event (per the telemetry schema), the comparison signals derived from content-aware analysis, and the quality proxies. None of those contain the content itself.
- **User reaction proxies apply only to the production run.** The customer reacted to the production output, not the shadow — so accept rate, edit distance, and survival are properties of the production run only. The shadow gets completion proxies, structural proxies, and the direct-comparison signals (how it looked relative to production). This is a real limitation of shadow mode as a measurement mechanism: it can tell Etak whether a candidate is *stable*, *structurally reasonable*, and *substantively comparable* to production on real inputs, but it cannot tell Etak whether customers would *prefer* it. That second question only experimentation can answer.
- **Shadow runs are sampled, not exhaustive.** Running two models for every request doubles inference cost. Shadow doesn't have to hit 100% of traffic — a 1% or 5% sample is usually enough to detect stability issues or structural divergence, and it keeps the cost impact bounded. The sampling rate is a per-shadow configuration, with defaults that favor cost-consciousness.
- **Shadow runs are latency-shielded.** The shadow execution runs asynchronously to the production path. The customer's response time is the production run's latency, full stop. If the shadow takes twice as long, that's data for the analysis, but it doesn't delay the customer. The gateway fires the shadow and doesn't wait on it.
- **Guardrails apply to shadow runs too.** If a shadow run hits an error, exhausts a cost ceiling, or produces structurally bad output, that's logged and aggregated. Shadow candidates with high error rates get flagged and blocked from further sampling automatically — the same rollback logic the experimentation harness uses, but applied to the shadow's accumulating signal rather than to customer exposure.
- **Shadow results produce a go/no-go signal for experimentation eligibility.** After enough shadow runs accumulate for a candidate (the "enough" is a function of task volume and signal stability), the system produces a summary: "this candidate is stable enough to run in an experiment," "this candidate has a higher error rate than production on live traffic and shouldn't be experimented on until that's fixed," "this candidate diverges substantively from production in X% of cases — worth a closer look before experimentation," or "inconclusive, needs more samples." The experimentation harness reads that signal before accepting a candidate into a live experiment.

## Why This Could Work

- Shadow mode is a well-established pattern for deploying risky code in non-AI contexts (database migrations, service rewrites, network routing changes). The AI-specific twist is that the shadow *output* is also interesting, not just the shadow *behavior*. Borrowing the pattern from general infrastructure and adapting it to agent execution is a natural move, not an invention.
- The data ownership stance, sharpened to distinguish retention from processing, actively permits content-aware comparison. This means shadow mode can extract its most valuable signal — "how similar was the candidate's output to production's on this real customer input?" — without any new privacy compromise. The signal Etak gets from shadow is genuinely rich because the constraint is where it belongs (on what persists), not where it would have been crippling (on what can be analyzed in the moment).
- The cost of shadow mode is bounded by the sampling rate, and the sampling rate can be tuned per-candidate. Low-confidence candidates (early-stage, large architectural changes) can be shadowed at higher rates; incremental tweaks can be shadowed at lower rates or skipped entirely. The cost is under explicit control rather than being a blanket tax.
- Shadow mode doesn't require any of the sophisticated infrastructure of the experimentation harness (cohorting, statistical power, rollback, disclosure). It's simpler to build, and it can ship before the harness does — providing some of the safety benefit even while the full experimentation story is still being assembled.

## Open Questions

- **What fraction of traffic should shadow by default?** Too little and the signal is weak; too much and the cost doubles unnecessarily. The right answer probably depends on the task's traffic volume — high-volume tasks can shadow at 1% and still get meaningful signal, low-volume tasks may need 10% or more. This needs to be a per-task configuration, not a global constant.
- **What's the right comparison method?** Options include structural metrics (token counts, code block counts), embedding similarity over outputs, LLM-as-judge comparing both outputs against the request, or some combination. Each has different cost, signal strength, and noise characteristics. The right answer is probably multiple methods running in parallel with the analysis layer picking the most informative signal per case — but that needs to be validated.
- **What about tasks with real side effects?** If a task-agent takes actions — writes to a database, makes an API call, creates an artifact — running it twice produces side effects twice. Shadow mode has to handle this: either the shadow run is sandboxed (tool calls are mocked or logged but not executed), or certain tasks are excluded from shadowing. Sandboxing is doable but changes the fidelity of the shadow signal, because "how the candidate behaves with real tool results" is part of what you want to measure.
- **Do customers need to be told shadow mode exists?** Under the sharpened data ownership stance, shadow mode is not a new category of data handling — it's processing that doesn't retain. That puts it in the same category as "the production agent runs internal candidates and picks the best," which customers generally aren't told about. The conservative answer is still to disclose the existence of shadow mode in whatever document describes how Etak processes customer data, so the customer has the information if they want it, but per-request consent is not needed. (This depends on `customers-accept-ephemeral-processing` holding — see that assumption.)
- **How long do shadow samples need to accumulate before the go/no-go is meaningful?** Related to the experimentation power problem — there's a minimum sample size below which shadow mode can't tell you anything. The system should know its own limits and refuse to produce confident signals below them.
- **What happens when the shadow candidate fails in interesting ways?** If a shadow run produces a bizarre output that (had it been served) would have been notable, is that a pattern for the gap-routing system (idea I)? Shadow runs never got flagged by humans — there was no human to flag them — so the routing pipeline doesn't naturally apply. But the failure is still signal. The system needs a path for shadow failures to reach the team even without a human in the loop.
- **Does shadow mode ever get used in reverse?** Is there a case for running *production* as a shadow against a *candidate* — where the candidate is already being shipped to customers but production is still running in parallel as a rollback safety net? This is a different pattern (hot-standby) and probably belongs in a different idea, but worth considering whether the shadow runner infrastructure could support it.
- **How does shadow interact with cost accounting?** If shadow runs are billed to "R&D" while production runs are billed to customer usage, who pays for the shadow? This is a bookkeeping question that affects how the cost control is modeled, and eventually how the team decides whether a given experiment is worth the shadow budget it consumes.
