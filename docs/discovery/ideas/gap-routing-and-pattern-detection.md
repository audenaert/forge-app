## Description

The downstream half of the human-observed-gap loop. In-flow gap capture (idea H) gets gaps *into* the system; this idea is about what happens to them once they're there — how they get classified, how they get routed to somebody who can act on them, how patterns across many flags surface to the team, how a gap becomes a decision to change something in the framework, and how the loop visibly closes back to the user who flagged it.

Conceptually, a flagged gap enters a pipeline with four stages:

1. **Ingestion and enrichment.** The raw flag (structural context from the run, optional user text, optional content share, category tag) lands in a durable queue. Enrichment attaches everything that can be inferred automatically: which task-agent version produced the run, what experiment cohort the customer was in, what the quality proxy score was, how this flag compares to the customer's historical flag rate, whether similar flags exist from other users in the same time window.
2. **Classification and routing.** The flag is classified — is it a prompt problem, a model problem, a tool-call problem, a data freshness problem, a docs gap, a UX problem, a genuinely novel feature request? Classification decides where the flag goes: to the owner of a specific task-agent, to an infrastructure team, to UX, to docs, or to an inbox for things that don't fit the existing categories. This stage is the hardest part of the system and the part that matures the most over time — early versions are heavily manual, later versions lean on automation.
3. **Pattern detection.** Individual flags are signal; clusters of flags are direction. The system surfaces "twelve customers flagged the test-plan agent this month with variations of 'missed the point'" as a first-class observable, distinct from "here's a list of twelve individual flags." Patterns are what turn flagging from a bug-report system into a learning system.
4. **Resolution and closure.** Flagged gaps get resolved in some form — a new task-agent version, a skill refinement, a docs update, a deliberate "we heard you and won't change this" decision, or "we can't act on this yet but it's tracked." Whatever the resolution, the user who flagged it eventually finds out what happened. If the loop doesn't close visibly, the upstream capture mechanism decays.

The pipeline is deliberately humble in its early form. Most of the "routing" is a human looking at a dashboard. Most of the "classification" is that same human assigning tags. Most of the "pattern detection" is visual inspection of a list grouped by task-agent. The point of framing all four stages now is to make sure the *shape* is right — that the system has a home for each concern — so that the automation can be added to each stage incrementally as volume grows.

## Strategic Rationale

The routing and pattern-detection system is what turns raw flags into framework improvements. Without it:

- **Flags pile up and nothing happens.** Every feedback system that doesn't have a credible downstream path ends up as a graveyard. The capture mechanism keeps running for a while out of habit and then collapses when users notice nothing ever changes. Building the downstream *before* the capture would be premature, but building them too far apart is fatal.
- **Signal is lost in noise.** Twelve flags on the same task-agent in a week are a huge signal. Twelve flags mixed in with three hundred across all agents are invisible. Without pattern detection, the team sees "a lot of feedback" and reacts by sampling — which is exactly the bias the learning loop is supposed to eliminate.
- **Ownership is ambiguous.** A flag that mentions a bad test plan could belong to the test-plan agent author, the prompt infrastructure team, the docs writer who documented the flow wrong, or nobody. Without routing, it defaults to "somebody else," which is to say nobody, which is to say the flag dies.
- **The learning loop has no bridge to the discovery graph.** The discovery graph (objectives, opportunities, ideas, assumptions) is where Etak thinks about what to build. Flagged gaps are evidence about whether what was built is working. Without a bridge, the graph and the flags live in different worlds — and the graph stays abstract while the flags stay operational. The routing system is where a pattern of flags can become an input to the graph (a new opportunity, an assumption about an existing idea, a critique of a shipped idea).

The subtle rationale: the *existence* of a working routing and pattern-detection system changes what kinds of gaps users are willing to report. Users calibrate their flagging rate against their perception of whether flagging matters. A system where flags visibly produce improvements raises the flagging rate over time; a system where flags disappear lowers it. The downstream system is, indirectly, the thing that determines how much upstream signal exists at all.

## How It Could Work

- **The queue is the telemetry stream.** Flag events from idea H land in the same event stream as run, reaction, and survival events (per idea C), just with a distinct event type. This means the routing system reads from infrastructure that already exists, and flag data is joined to run data for free.
- **Enrichment is a read-only workflow, not a write.** When a flag is ingested, a background job computes everything the system can infer automatically — task-agent version, cohort, quality proxy score, related flags within a time window — and stores the enriched record. The raw flag is never modified. This preserves provenance and makes the enrichment logic auditable and re-runnable (if the enrichment rules change, old flags can be re-enriched).
- **Classification starts manual and gets assisted.** Early on, a human (initially Neal, eventually a team member) opens flagged gaps in a triage UI and assigns category tags, ownership, and status. Over time, the system learns from those human decisions and starts proposing classifications — "this looks like a prompt issue on the test-plan agent based on 14 similar flags." The human still decides, but the machine-proposed classification cuts the per-flag overhead substantially. Fully automated classification is a later step, and one that should always retain human override.
- **Routing maps to the task-agent registry.** When a flag is classified as "about task-agent X," the routing system looks up the owner of task-agent X in the registry (idea B) and assigns the flag to them. When the classification is structural ("this is a gateway timeout, not an agent problem"), the routing goes to the relevant infrastructure team. When there's no clear owner, the flag goes to a shared inbox — not to purgatory.
- **Pattern detection has two layers.** A coarse layer (simple groupings: flags per task-agent per week, flags per experiment cohort, flags per customer) is built with straightforward warehouse queries and surfaces on a dashboard. A fine layer — clustering semantically similar flags, detecting when a new pattern is emerging that isn't just a known old pattern — uses embedding similarity over the flag text and category tags. The fine layer is only available for flags where the user opted to share text or content; the coarse layer is available for everything.
- **Content-aware analysis is opt-in and scoped.** Flags where the user granted content permission (per idea H) can be analyzed more deeply — similarity against other shared flags, root-cause hypotheses from an LLM summarizer, sample quotes in the dashboard. Flags without content permission still participate in coarse patterns but not in content-aware analysis. The distinction is enforced at the data layer, not trusted to the people running the analysis.
- **Resolution is a status, not a message board.** A flagged gap has a status — new, triaged, in progress, resolved, closed as won't-fix, closed as duplicate — that changes deliberately via the triage UI. The status is visible to the user who flagged it. Status changes produce notification events that the user sees in their account, email, or wherever they've opted to hear from Etak.
- **Loop closure is the top priority feature, not an afterthought.** When a gap is resolved, the user who flagged it is told, specifically, what happened. "The test-plan agent has been updated; your flag contributed to the change." Generic acknowledgements ("we appreciate your feedback") are worse than silence. Specific acknowledgements — even terse ones — are the single thing that keeps the upstream mechanism alive.
- **Patterns feed back into the discovery graph.** When a pattern across many flags crystallizes into "something we should think about at the product level," the routing system can create (or suggest) a discovery graph node — an opportunity, an assumption against an existing idea, a critique. The flag cluster becomes the evidence attached to the graph node. This is how operational signal becomes strategic input without a human having to remember to make the connection.
- **The triage UI is itself a task-agent.** In keeping with the dogfood principle, the triage interface is built on top of the same task-agent registry it's surfacing flags about. A "triage flags" task-agent proposes classifications, suggests ownership, identifies duplicates. When *that* agent gets flagged for bad proposals, the same loop fixes it. Recursive but apt.

## Why This Could Work

- The hard parts of this system are pattern detection and classification, and both benefit from starting manual and getting automated over time. The manual version ships early and is immediately useful at dogfood scale (one person reviewing a handful of flags a day). Automation is added when volume demands it, and by then there's enough history to train the automation against.
- The integration points already exist conceptually. The task-agent registry (B) is where routing sends ownership. The telemetry stream (C) is where flags live. The quality proxy model (D) is where flag signal can be incorporated. The pipeline isn't new infrastructure — it's the workflow that makes the existing infrastructure produce decisions.
- Every developer-tool company with a mature feedback culture ends up with something structurally similar (Linear's bug triage, GitHub's issue triage, Cursor's user research pipeline). Etak is assembling the pieces deliberately rather than letting them emerge by accident, and getting them wired into the learning loop from day one.
- The discovery-graph integration is unusual and potentially a real differentiator. Most products treat operational feedback (bugs, complaints) and strategic discovery (opportunities, ideas) as separate concerns handled by separate systems. Etak's opportunity is to make them the same concern at different altitudes, with flags being the lowest-altitude view and graph nodes being the highest.

## Open Questions

- **How much classification can be automated before it hurts?** LLM-assisted classification scales well but can produce confident-wrong results that poison the routing. Pure-manual classification doesn't scale. The right answer is probably a human-in-the-loop where the machine proposes and the human confirms, but the confidence threshold for promoting classifications to "automatic" is a judgment call.
- **How is duplicate detection done?** If the same underlying problem produces twenty similar flags in a week, the team shouldn't see twenty separate items — they should see one cluster with twenty instances. But "similar" is hard to define precisely, and the wrong deduplication rule (too aggressive) loses individual signal while the right one (too conservative) drowns the dashboard. Start simple and tune; don't over-engineer.
- **How fast is the SLA for loop closure?** Users flag things expecting *some* response on *some* timescale. The worst outcome is "never hear back"; the second-worst is "hear back six months later in a way that suggests nothing happened." The team needs a default cadence — maybe "every flag gets a status update within a week, even if the status is just 'triaged, waiting on investigation'" — that it can actually keep.
- **Who sees the dashboard?** Internal team? Individual agent authors? Customers? Each extension has costs and benefits. The default probably starts as "internal team only" and opens up over time as trust grows, but it's worth thinking about the eventual shape now so the data model can support it.
- **How is the discovery-graph bridge built without creating noise there?** Every flag pattern could become a graph node, but that would quickly clutter the discovery graph with low-altitude issues. There has to be a threshold — "a pattern becomes a graph node when it rises above X severity or Y frequency" — and the rule for that threshold is a product decision, not a technical one.
- **What happens to flags on experimental variants?** A flag on a variant that was rolled back tells us something different from a flag on the production variant. The routing system needs to know about experimentation context and treat the two cases differently — a rolled-back variant flag may be expected behavior, not a bug.
- **How are flag rates baselined?** Without a baseline, "flag rate up 20% this week" is not interpretable — up from what? The system needs rolling baselines per task-agent so that spikes are distinguishable from background noise, and so that the system can alert the team when something suddenly gets worse instead of waiting for a human to notice.
- **How does the system avoid reinforcing existing team priorities?** If the team routes flags toward the agents they were going to work on anyway, the routing becomes a self-fulfilling prophecy — only the pre-prioritized agents get attention, and the rest decay silently. Some structural bias toward "give under-attended agents a second look" may be needed.
- **What's the escalation path for high-severity patterns?** A sudden spike of flags that all mention security concerns, or data leakage, or offensive outputs, is not something that can wait for the normal triage cadence. The system needs an alerting layer that can wake somebody up for patterns matching pre-defined severity criteria — without false-alarming so often that alerts get muted.
